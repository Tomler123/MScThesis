## Optimized Implementation

```python
from collections import defaultdict

def group_anagrams(words: list[str]) -> list[list[str]]:
    """
    Group words that are anagrams of each other.

    Each group preserves the original input order of its words, and groups
    are ordered by the first appearance of their key in the input.

    Args:
        words: A list of strings to group by anagram equivalence.

    Returns:
        A list of groups, where each group is a list of anagram-equivalent words.

    Time complexity:  O(n * k log k) — n words, each sorted in O(k log k).
    Space complexity: O(n * k)       — storing all words in the hash map.
    """
    groups: dict[tuple, list[str]] = {}
    for word in words:
        key = tuple(sorted(word))       # O(k log k) per word
        if key not in groups:
            groups[key] = []
        groups[key].append(word)
    return list(groups.values())
```

***

## Why the brute-force is slow

The brute-force uses a **nested loop**: for every word `i`, it scans all remaining words `j > i` and compares `sorted(words[i]) == sorted(words[j])`. That sort is O(k log k) per comparison, and there are O(n²) pairs, giving **O(n² · k log k)** overall. For 10 000 words it performs ~50 million comparisons; for 100 000 words, ~5 billion — completely impractical.

## Data structure used and why

The key insight is that **two words are anagrams if and only if their sorted-character tuples are identical**. A plain Python `dict` maps each such sorted tuple → list of original words. Because dict lookup and insertion are O(1) average, you only ever touch each word *once* — no pairwise comparison needed at all. A `tuple` is used instead of a `str` only because `tuple(sorted(word))` is the most direct expression; a `"".join(sorted(word))` string key works equally well.

Insertion order of a Python `dict` is guaranteed since Python 3.7, so:
- **Group order** is automatically the order of first encounter.
- **Within-group order** is preserved because each word is simply appended.

## Time & space complexity

| | Brute-force | Optimized |
|---|---|---|
| **Time** | O(n² · k log k) | **O(n · k log k)** |
| **Space** | O(n) (only the `used` flags + output) | O(n · k) (the hash map + output) |

The optimized version trades a small amount of extra memory (storing all keys in the map) for a dramatic reduction in time — the outer O(n²) loop collapses into a single O(n) pass, with one O(k log k) sort per word.
