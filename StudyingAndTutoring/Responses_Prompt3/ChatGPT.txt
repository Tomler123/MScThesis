## 1. Key Ideas

**Big-O notation** describes how the runtime of an algorithm grows as the input size (n) becomes large. It focuses on the **worst-case growth rate** of the number of operations.

If an algorithm takes longer and longer as (n) increases, Big-O tells us the general pattern of that growth.

### Formal definition

We say

[
f(n) = O(g(n))
]

if there exist positive constants (c) and (n_0) such that for all (n \ge n_0),

[
f(n) \le c \cdot g(n)
]

This means that, from some point onward, (f(n)) does not grow faster than a constant multiple of (g(n)).

### Common complexity classes

Listed from smaller growth to larger growth:

[
O(1),\quad O(\log n),\quad O(n),\quad O(n \log n),\quad O(n^2),\quad O(2^n)
]

What they mean:

* **(O(1))**: constant time. Runtime does not depend on (n).
* **(O(\log n))**: grows very slowly. Common in binary search.
* **(O(n))**: grows proportionally with input size.
* **(O(n \log n))**: slightly worse than linear. Common in efficient sorting.
* **(O(n^2))**: often appears with nested loops over the same data.
* **(O(2^n))**: extremely fast growth. Common in brute-force recursion over subsets.

### Why we drop constant factors and lower-order terms

Big-O is about **overall growth trend** for large (n), not exact runtime.

Examples:

* (3n = O(n))
* (n^2 + 5n + 7 = O(n^2))

Why?

* Constant factors like (3) matter less than the type of growth.
* Lower-order terms like (5n) become insignificant compared with (n^2) when (n) is large.

So asymptotic analysis keeps the dominant term and ignores the rest.

## 2. Worked Example

Analyze this pseudocode:

```text
for i from 1 to n:
    for j from i to n:
        print(i, j)
```

We want the total number of `print` operations.

### Step 1: Outer loop

The outer loop runs with

[
i = 1, 2, 3, \dots, n
]

So it runs **(n) times**.

### Step 2: Inner loop for each fixed (i)

For a given (i), the inner loop runs from (j=i) to (j=n).

That gives:

* when (i=1): (n) prints
* when (i=2): (n-1) prints
* when (i=3): (n-2) prints
* ...
* when (i=n): (1) print

So the total number of prints is

[
n + (n-1) + (n-2) + \cdots + 1
]

### Step 3: Sum the series

This is the sum of the first (n) positive integers:

[
1 + 2 + 3 + \cdots + n = \frac{n(n+1)}{2}
]

So the exact number of `print` operations is

[
\frac{n(n+1)}{2}
]

### Step 4: Simplify to Big-O

Expand:

[
\frac{n(n+1)}{2} = \frac{n^2+n}{2}
]

Ignoring the constant factor (\frac{1}{2}) and the lower-order term (n), the dominant term is (n^2).

Therefore, the overall time complexity is

[
O(n^2)
]

### Conclusion

* Exact number of `print` operations:

[
\frac{n(n+1)}{2}
]

* Asymptotic time complexity:

[
O(n^2)
]

## 3. Practice Exercises (with solutions)

### Exercise 1

```text
count = 0
for i from 1 to n:
    count = count + 1
```

**Solution:**

The loop runs exactly (n) times.

Each iteration does constant work, so total work is proportional to (n):

[
T(n) = cn
]

for some constant (c). Therefore,

[
T(n) = O(n)
]

**Answer:** (\boxed{O(n)})

---

### Exercise 2

```text
for i from 1 to n:
    for j from 1 to n:
        print(i, j)
```

**Solution:**

* Outer loop runs (n) times.
* For each outer iteration, the inner loop also runs (n) times.

So total number of `print` operations is

[
n \cdot n = n^2
]

Thus the total runtime grows proportionally to (n^2).

[
T(n) = O(n^2)
]

**Answer:** (\boxed{O(n^2)})
