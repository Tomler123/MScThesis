Here is the implementation of the `merge_intervals` function followed by the requested complexity analysis.

```python
def merge_intervals(intervals: list[tuple[int, int]]) -> list[tuple[int, int]]:
    """
    Merge all overlapping intervals and return a sorted list of non-overlapping intervals.
    
    Args:
        intervals: A list of tuples, where each tuple represents (start, end).
        
    Returns:
        A new list of merged and sorted tuples.
    """
    if not intervals:
        return []

    # 1. Sort the intervals based on the start time.
    # This ensures that any potential overlaps are adjacent in the list.
    sorted_intervals = sorted(intervals, key=lambda x: x[0])

    # Initialize the merged list with the first interval.
    merged = [sorted_intervals[0]]

    for current in sorted_intervals[1:]:
        last_start, last_end = merged[-1]
        curr_start, curr_end = current

        # 2. Check for overlap:
        # Since they are sorted by start time, we only need to check if 
        # the current start is less than or equal to the previous end.
        if curr_start <= last_end:
            # Overlap found: Update the end of the last interval in 'merged'
            # to be the maximum of the two ends (handles nested intervals).
            merged[-1] = (last_start, max(last_end, curr_end))
        else:
            # No overlap: Add the current interval as a new entry.
            merged.append(current)

    return merged
```

---

### Complexity Analysis

#### Time Complexity: $O(n \log n)$
* **Sorting:** The primary bottleneck is the sorting step. Sorting $n$ intervals using Python's Timsort takes $O(n \log n)$ time.
* **Merging:** Once the intervals are sorted, we perform a single linear pass through the list to merge them. This takes $O(n)$ time.
* **Total:** $O(n \log n) + O(n)$ simplifies to **$O(n \log n)$**.

#### Space Complexity: $O(n)$
* In the worst case (where no intervals overlap), the `merged` list will contain all $n$ intervals. Additionally, the `sorted()` function creates a new list of size $n$.

---

### Why sorting is essential

Sorting by the **start time** is the crucial step that makes an efficient solution possible for two main reasons:

1.  **Localizing Overlaps:** When intervals are sorted by start time, any interval that could potentially overlap with interval $A$ must appear either immediately before or after $A$ in the list. This transforms a global search problem (checking every interval against every other interval, which is $O(n^2)$) into a local linear scan.
2.  **Order of Processing:** By processing intervals in chronological order of their start times, we can make a definitive decision about the current "active" interval. We know that no subsequent interval in the list will start *earlier* than the one we are currently looking at. This allows us to simply maintain a "last merged" interval and update its endpoint as we find overlaps, knowing we never have to "look back" at previous intervals.
